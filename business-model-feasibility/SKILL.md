# 商业模式可行性验证 — 导入期的核心任务

## R — 原文 (Reading)

> 商业模式可行性 = 销售可行性（需求是否真实存在）+ 利润可行性（UE模型能否跑正）。时间对标和空间对标是验证需求真实性的两大工具——"人类的底层需求是相通的，成熟市场的经验可作参考"。
>
> — 肖璟, 第2章

---

## I — 方法论骨架 (Interpretation)

导入期的核心问题是"这个商业模式能不能跑通"。可行性验证分为两道关卡：

### 第一关：销售可行性（需求验证）

核心问题是**"用户真的有这个需求吗？"**。三种验证方法：

| 验证方法 | 操作 | 底层逻辑 |
|---------|------|----------|
| **时间对标** | 回顾历史，看过去有没有类似需求被成功满足过 | 人性需求不变 |
| **空间对标** | 对标成熟市场（美/日/欧），看类似模式是否成功过 | 人类底层需求相通 |
| **创新层级判断** | 检查该商业模式依赖的前置技术是否已经成熟 | 底层技术是上层创新的前提 |

### 第二关：利润可行性（UE模型验证）

核心问题是**"以最小运作单位来看，能赚钱吗？"**。UE模型 = 单位经济模型，以最小运作单位（单店/单车/单品/单客）的收入成本结构来验证利润可行性。

**关键原则：不只搭建当前UE模型，必须搭建三版本**：

```
当前UE模型 → 盈亏平衡UE模型 → 终局UE模型
（现状）     （何时能赚钱？）    （稳态能赚多少？）
```

**UE模型核心指标**：
- 获客成本（CAC）
- 用户生命周期价值（LTV）
- 单位毛利率
- 固定成本摊销比例
- 盈亏平衡点

### 第三道防线：调头能力评估

当商业模式走不通时，团队/企业是否有能力快速转型？评估要点：
- 团队技能的多样性（是否只能做这一件事？）
- 资金储备是否允许试错（有多少次"调头"机会？）
- 现有资产是否可以复用（技术/渠道/品牌能否用于新方向？）

### 商业模式画布（概览工具）

| 要素 | 内容 |
|------|------|
| 价值主张 | 解决了什么核心问题？ |
| 客户细分 | 目标用户是谁？付费意愿如何？ |
| 收入来源 | 谁付费？怎么付？频率？ |
| 成本结构 | 固定成本 vs 变动成本？规模效应在哪？ |
| 核心资源 | 什么是不可或缺的？ |

## A1 — 书中的应用 (Past Application)

### 案例 1: C2C模式的销售可行性验证
- **问题**: 中国第一代互联网创业照搬美国C2C模式（百度→Google、淘宝→eBay），如何判断是否可行？
- **方法论的使用**: 空间对标——美国模式成功的前提（信用体系、物流基建、支付习惯）在中国是否具备？时间对标——历史上中国的集市/跳蚤市场的C2C交易是怎么运作的？
- **结论**: C2C需求本身是真实的（买卖双方都有对接需求），但照搬美国模式的可行性低——需要本地化改造（如淘宝的担保交易）
- **结果**: 纯C2C照搬者多失败，本地化改造者成功

### 案例 2: 共享篮球——需求伪命题
- **问题**: 2017年共享篮球项目，扫码租篮球，用后归还
- **方法论的使用**: 时间对标——"打篮球忘了带球"这个需求历史上是怎么解决的？借朋友/球场边小店租球，成本极低。共享篮球的押金+扫码流程反而更复杂
- **结论**: 销售可行性不成立——历史上低成本方案已充分满足需求，共享方案没有创造增量价值
- **结果**: 项目迅速失败

### 案例 3: 恒大冰泉——砸钱绕不过技术诀窍
- **问题**: 恒大冰泉20天砸13亿广告费，设定100亿销售额目标，能成功吗？
- **方法论的使用**: 利润可行性验证——包装饮用水行业的核心know-how（对水的研究、水源布局、渠道管理）无法通过资本快速复制。农夫山泉创始人评价"没有知识积累，光砸钱没有用"
- **结论**: 资本可以买渠道和广告，但买不到技术诀窍——UE模型中的关键成本项（获客成本/渠道维护成本）在砸钱模式下会严重超标
- **结果**: 3年巨亏40亿，最终出售

## E — 可执行步骤 (Execution)

1. **验证销售可行性**
   - 时间对标：抽象底层需求 → 搜索历史上满足类似需求的方案 → 比较新旧方案的优劣
   - 空间对标：选择成熟参照市场 → 搜索该市场对应模式的成败
   - 完成标准: 产出"需求真实/伪需求/需进一步验证"的判断 + 对标依据

2. **搭建三版本UE模型**
   - 当前UE：基于公开数据（客单价、毛利率、租金、人工等）
   - 盈亏平衡UE：关键拐点变量是什么？需要什么条件？
   - 终局UE：对标成熟市场/成熟企业的稳态UE
   - 完成标准: 至少产出当前UE和盈亏平衡UE，标注"终局UE为推测"
   - 搜索执行参考: `reference/search-specs.md` 的导入期搜索指引
   - 数据标注参考: `reference/data-annotation-standards.md`

3. **评估创新层级依赖**
   - 检查该商业模式依赖的前置技术处于什么成熟度（参考Gartner曲线）
   - 完成标准: 列出所有关键技术依赖 + 每项的成熟度判断

4. **评估调头能力**（如适用）
   - 团队技能多样性 / 资金试错空间 / 资产可复用性
   - 完成标准: 给出"调头能力强/中/弱"的判断

## B — 边界 (Boundary) ★

### 不要在以下情况使用此 skill

- 行业已处于成长期或成熟期——商业模式已被市场验证，可行性分析的价值边际递减
- 评估纯技术创新（如新药研发）——药物可行性取决于临床试验而非商业模式
- 用户只需要了解商业模式的**描述**而非**验证**——描述商业模式只需要画布，验证需要数据和推理

### 作者在书中警告的失败模式

- **静态UE模型陷阱**：只用当前UE模型判断就否定一个行业——必须搭建三版本UE（当前→盈亏平衡→终局），关键看"有没有通向终局的路径"
- **资本可绕过≠所有壁垒可绕过**：砸钱能绕过劳动力、土地位置、资本——但绕不过技术诀窍/Know-how、网络效应、数据积累
- **渠道催熟的质量问题**：瑞幸5年开万店，但商务拓展团队虚报租金、劣质选址，2021年关店600余家——渠道可被资本"催熟"但质量极差

### 作者的盲点 / 时代局限 (来自 BOOK_OVERVIEW.md)

- **大行业偏好**：UE模型分析默认市场规模足够大、值得投入时间搭建。对于小而美的利基市场，三版本UE模型的精细度可以适度降低
- **精英信息环境**：终局UE模型需要大量行业内部数据（如竞品的真实毛利率），普通研究者可能难以获取
